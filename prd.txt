1. Executive Summary 
Lighthouse and similar lab tools are synthetic — they simulate a page load in a controlled 
environment. They cannot capture what happens when a real user scrolls, clicks, or navigates 
between routes in a live React application. This is the gap react-perf-hud fills. 
 
react-perf-hud is a draggable, in-app HUD component that attaches PerformanceObserver listeners 
to the live browser session and surfaces Core Web Vitals (LCP, INP, CLS) alongside diagnostic 
signals (FCP, TTFB, TBT) in real time. It is designed as a zero-config import that disappears 
completely from production builds. 
 
The SWOT analysis identified five strategic priorities that directly shape this PRD: 
• Leverage the real-user visibility gap as the primary value proposition (Strength S1) 
• Resolve browser compatibility silently and honestly — no false metrics (Weakness W2) 
• Make production safety non-negotiable, not a footnote (Weakness W1) 
• Design the beacon schema upfront so this can grow into a lightweight RUM tool (Opportunity 
O3) 
• Build a pluggable metric registry to survive CWV spec churn (Threat T3)

2. Problem Statement 
2.1  The Lab vs. Field Divide 
Lab tools such as Lighthouse, WebPageTest, and Chrome DevTools Performance panel measure a 
synthetic, reproducible page load. They are invaluable for auditing but structurally incapable of 
capturing: 
• INP (Interaction to Next Paint) on real user gestures within complex component trees 
• CLS accumulation across SPA route transitions and lazy-loaded content 
• LCP for content rendered after hydration or after API calls resolve 
• Long-task blocking time under real CPU contention, not simulated throttle 
 
Field data from tools like Google Search Console represents aggregated RUM but is delayed by 
days and provides no in-session debugging capability. 
2.2  The Developer Experience Gap 
There is no first-class React-native tool that lets a developer drop one component into a staging 
environment and immediately see live vitals alongside the UI they are testing. Existing options 
require: 
• Chrome extension installs (not portable, not reproducible in CI) 
• Manual PerformanceObserver scripting with no visualisation 
• Full RUM platform integration (Datadog, New Relic) — expensive and complex for early-stage 
debugging 
 
 
3. Goals & Non-Goals 
3.1  Goals 
• Provide real-time, in-session Core Web Vitals with Good/Needs Improvement/Poor ratings 
embedded in the UI 
• Require exactly one line of JSX to enable — no configuration mandatory 
• Guarantee zero production bundle impact through build-time elimination 
• Support optional beaconing of metric data to any analytics endpoint 
• Degrade gracefully and honestly on non-Chromium browsers 
• Architect the metric system to be extensible as CWV specs evolve 
3.2  Non-Goals 
• This is NOT a replacement for full RUM platforms (Datadog, New Relic, SpeedCurve) 
• This is NOT a server-side performance tool — it operates in the browser only 
• This is NOT designed for production end-user exposure — only dev/staging/CI

• This does NOT provide historical trend analysis or p75/p99 aggregation out of the box

6. Product Features 
SWOT Basis: S2, S3 — Plug-and-play DX; S3 — Draggable HUD; W5 — Embedded thresholds; O1 — npm 
publishability 
6.1  Draggable HUD 
• Rendered as a fixed-position overlay using a React portal 
• Drag handle using pointer events, clamped to viewport bounds 
• Position persisted in sessionStorage so it survives hot reloads 
• Collapse mode: shrinks to a 40×40 px icon pill when minimised 
• Supports dark / light / auto themes via CSS custom properties 
 
Visual anatomy of each metric tile: 
◦ Metric name and current live value 
◦ Delta indicator (arrow + value change from previous reading) 
◦ Good / Needs Improvement / Poor badge using official CWV colour coding 
◦ Inline sparkline of last 20 readings rendered on an HTML canvas

6.3  Production Safety 
SWOT Basis: W1 — Accidental production inclusion is the highest-severity weakness 
Production safety is enforced at three independent layers: 
• Runtime guard: component returns null immediately if process.env.NODE_ENV === 'production' 
• Build-time strip: a provided Babel/SWC plugin eliminates all import and JSX references at 
compile time — zero bytes in the production bundle 
• Type-level hint: the TypeScript type signature carries a @devOnly JSDoc tag, surfaced as a lint 
warning by a provided ESLint rule if the component is detected in a non-development import 
path 
All three layers are independent so that any single layer is sufficient protection. The Babel plugin is 
the recommended primary path.

7. SSR & React Server Component Awareness 
SWOT Basis: W4 — SSR/RSC blind spots identified in SWOT 
Client-side PerformanceObserver cannot measure server execution time. The following approach 
bridges the gap: 
• TTFB is captured via PerformanceNavigationTiming.responseStart, available on the client but 
reflecting server response latency 
• SSR frameworks (Next.js, Remix) may inject Server-Timing response headers. When present, 
react-perf-hud reads these via a <meta name='server-timing'> tag and surfaces them as 
supplemental tiles labelled SERVER 
• Hydration delta is measured by wrapping the first client-side paint after document load — the 
gap between FCP and TTI serves as a hydration cost proxy 
• RSC payload delivery time is captured via PerformanceResourceTiming on fetch requests 
matching /_rsc or __RSC_MANIFEST patterns (configurable via rsaPrefixes prop)

8.2  Pluggable Metric Registry 
SWOT Basis: T3 — CWV spec churn; T1 — Chrome DevTools parity threat 
Every metric is defined by a MetricDefinition interface, removing the need to modify core logic when 
specs change: 
• id: string — Metric identifier 
• observe: (cb) => () => void — Sets up and tears down the PerformanceObserver 
• extract: (entry) => number — Pulls the numeric value from an entry 
• thresholds: { good: number, poor: number } — Current CWV-aligned values 
• supportedBrowsers: BrowserSupport — Drives the compatibility matrix and '--' fallback 
react-perf-hud  |  Product Requirements Document  |  SWOT-Refined Page 7 
Adding a new metric requires creating one file implementing MetricDefinition and registering it. No 
changes to the HUD component or core engine.

12. Open Questions 
The following decisions require team alignment before Phase 01 begins: 
• Should budgets prop accept absolute values or percentile targets (p75)? Absolute is simpler; 
percentiles require a data aggregation layer not in scope for v1. 
• Should the HUD be draggable by default or locked by default? Draggable by default feels more 
useful but adds pointer-event complexity. Consider locked default with a drag prop. 
• Should @react-perf-hud/core be a peer dependency or bundled? Bundling simplifies the install; 
a peer dependency enables tree-shaking for consumers who only want the observer engine. 
• Should the AI recommendations layer (Opportunity O5) be scoped into v1 as a static rule engine, 
or deferred entirely to v2? Static rules (no external API call) add value with low complexity.
